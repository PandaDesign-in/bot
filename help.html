🐼 PandaAI User Guide

PandaAI is a single-user, GitHub-native PWA for architectural and interior design analysis. Load any 3D or 2D file, apply one or more AI expert personas, run mathematical analysis, and generate encrypted reports — all inside the browser with zero server infrastructure.

ℹ️ No installation needed. PandaAI runs entirely in your browser. All data is AES-256-GCM encrypted before leaving the browser. GitHub is the sole backend.

🚀 Quick Start

Three steps from zero to first analysis:

Enter your GitHub PAT
On first load, enter a GitHub Personal Access Token with repo scope. This is your single key — it authenticates the vault and derives the encryption key. Generate at GitHub → Settings → Developer settings → Personal access tokens. The PAT is stored in sessionStorage only and cleared when you close the tab.
Add your Groq API Key
Get a free key at console.groq.com (takes 30 seconds). Go to ⚙️ Settings → Groq API Key → Change and paste it. The key is encrypted and saved to your vault — you only enter it once. Without a Groq key, file loading and viewing still work but AI features are disabled.
Drop a file and analyse
Drag any 3D or 2D file onto the viewer, or click ＋ in the file tree. The format is detected automatically. Click Analyse File in the bottom bar to generate a full AI report using your selected persona(s).

🧮 The Math-er

The Math-er is PandaAI's dedicated computational geometry and cost quantification engine. It converts raw 3D geometry into precise BOQ-ready measurements, material quantities, and cost tables — all in ₹ INR using CPWD 2023 Delhi Schedule of Rates.

🧮 Key principle: The Math-er never guesses. Every value shows its working. Estimated values (from sampling) are clearly flagged as such. All costs are in ₹ INR only.

⚙️ How to Use The Math-er

Method 1 — Full Model Analysis (Analyse File button)

  1. Open the 👥 Personas panel (topbar)
  2. Check 🧮 The Math-er checkbox (can combine with other personas in Blend mode)
  3. Choose Generic or Expert depth
  4. Click Analyse File — the full model's geometry is sent to the Math-er
  5. Report appears in the analysis panel with cost table in ₹ INR
  6. Use refinement chat to ask: "Calculate cost if all concrete is M30" or "What if we use structural steel instead?"

Method 2 — Per-Mesh Analysis (Right-Click in 3D View)

Cost Calculation Method

The Math-er applies the following formula to generate a project estimate from geometry:

BOQ Cost Formula:

Material Quantity (from geometry) × CPWD SOR Rate (₹) = Line Cost
Sum of all line costs = Sub-Total
Sub-Total × 10% = Contingency
(Sub-Total + Contingency) × 15% = Overhead & Profit
Sub-Total + Contingency + O&P = Grand Total (₹ INR)

Complex geometry adds a Complexity Premium of 0–40% based on the Geometric Complexity Score.

Geometric Complexity Score → Premium

ScoreDescriptionPremium Applied
1–2Simple rectilinear geometry — standard box shapes, regular grid0%
3–4Moderate — some curves or irregular plan, mostly standard elements5–10%
5–6Complex — free-form surfaces, non-standard angles, compound curves10–20%
7–8Very complex — parametric/freeform geometry, unusual structural challenges20–35%
9–10Ultra-complex — sculptural, one-off, requires specialist subcontractors throughout35–40%

How Encryption Works

AES-256-GCM with PBKDF2 key derivation.
Your GitHub PAT → PBKDF2 (SHA-256, 310,000 iterations, 16-byte salt) → 256-bit AES key. Every file, report, and session is encrypted with a unique IV before any GitHub write. The PAT never leaves the browser. GitHub only ever sees ciphertext.

Offline Behaviour

The service worker pre-caches all modules, CDN libraries, and loaders on first install. After the first load in Tokyo (or any location), subsequent sessions run 100% from cache. The only network calls are Groq API and GitHub sync — nothing else.

Clear the module cache via ⚙️ Settings → Clear Module Cache. The app re-downloads everything on the next reload.

💡 Tips & Performance Notes

Large File Performance (200MB+)

FormatParsing strategyExpected behaviour
STL ≥ 50 MBWeb Worker (off main thread)UI stays fully responsive. Progress bar shows 0–100%. No stutter.
STL < 50 MBMain thread (synchronous)Brief pause (usually <2s on modern hardware)
FBX, GLTF, GLB (large)Main thread (Object URL)Toast warning shown. Browser may be unresponsive for 10–60s. GPU upload after parse may cause single-frame stutter.
IFC (BIM)web-ifc WASM (near-native)Fast for typical BIM models. Large complex IFC (>200 MB) may take 30–90s.

ℹ️ Memory note: A 200 MB STL file will use approximately 400–600 MB of browser RAM after parsing (positions + normals arrays + GPU upload). This is unavoidable for dense geometry. On machines with dedicated GPU (like RTX 4080 Super), GPU memory is used separately and does not affect browser RAM budget.

Best Practices

  • For Revit/Navisworks models: always export to IFC first — this gives the richest BIM data for analysis
  • For large assemblies: use GLB (binary GLTF) instead of FBX — faster parse, better material fidelity
  • For point clouds: down-sample to <10M points before uploading for best performance
  • For multi-part analysis: use Explode View first, then right-click individual parts for Math-er per-component cost analysis
  • Use Expert mode + single persona for the most technical, code-referenced reports
  • Use Blend mode (multiple personas) for comprehensive design reviews where you need multiple disciplines in one report
  • Save reports before closing the tab — encrypted reports are stored in your GitHub vault
